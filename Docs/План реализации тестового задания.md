## 1. Архитектура проекта и зависимости

### Шаг A1. Архитектурный подход и слои

Для тестового задания не нужно строить новый проект. Правильнее адаптировать текущую архитектуру `ShooterUpgradePrototype` под требования ТЗ.

Базовый подход:

- `Gameplay Layer`
  Игрок, враги, стрельба, урон, смерть, боевой цикл.
- `Progression Layer`
  Очки улучшения, уровни характеристик, применение апгрейдов, сохранение.
- `UI Layer`
  HUD, окно улучшений, синхронизация UI с runtime-данными.
- `Infrastructure Layer`
  Zenject, bootstrap, scene setup, resource loaders, storage.
- `Data Layer`
  ScriptableObject-конфиги статов, таблицы апгрейдов, enemy spawn config.

Текущий проект уже хорошо ложится на этот подход, потому что gameplay собран через `EntityMonoBehaviour`, `ComponentData` и `Action`, а инфраструктура уже разнесена по asmdef-модулям.

### Шаг A2. Пакеты и плагины

Ключевые технологии, которые уже есть и которые нужно сохранить:

- `Zenject`
  Используем как DI-контейнер для сервисов, storage, spawner, bootstrap и UI.
- `UniRx`
  Используем для реактивной синхронизации kill counter, HP, очков улучшения и UI.
- `EntityDataAction`
  Основной gameplay-подход: сущность = `ComponentData` + `Action`.
- `UI Toolkit`
  Основной visual backend для UI тестового задания.
- `UISystem`
  Оркестрация экранов, HUD и окон.
- `UIRootSystem`
  Корень UI и слои `HUD`, `Screens`, `Popups`.
- `Input System`
  Основной input backend.
- `Stats`
  Data-driven система характеристик.
- `WaypointsSystem` и `WaypointPathEntitySpawner`
  Готовый слой движения и спавна врагов.
- `AddressableLoaderSystem`
  Загрузка `UXML`, конфигов и других runtime-ресурсов.

### Шаг A3. Структура папок и ответственность модулей

Уже существующие модули:

- `Assets/Assemblies/AutoStrike`
  Основной gameplay-модуль. Здесь остаются игрок, враги, progression, UI и сервисы тестового задания.
- `Assets/Assemblies/SceneManagerIntergration`
  Bootstrap-точка входа и scene operations.
- `Packages/com.vladislavtsurikov.frameworks/UISystem`
  UI orchestration.
- `Packages/com.vladislavtsurikov.frameworks/UIRootSystem`
  Корень UI слоев.
- `Packages/com.vladislavtsurikov.frameworks/Stats`
  Характеристики и модификаторы.

Рекомендуемое логическое деление внутри `AutoStrike`:

- `Runtime/Player`
- `Runtime/Enemy`
- `Runtime/Progression`
- `Runtime/UI`
- `Runtime/Config`
- `Runtime/Infrastructure`

### Шаг A4. Критерии готовности архитектуры

Архитектура считается удачной, если:

- новые характеристики можно добавлять без переписывания core gameplay;
- UI не хранит бизнес-логику апгрейдов;
- враги, игрок и UI не завязаны друг на друга напрямую;
- конфиги лежат в `ScriptableObject`;
- runtime-сервисы создаются через DI;
- переход на mobile-input потребует замены input-слоя, а не переписывания игрока;
- battle UI живет в одном целевом pipeline.

## 2. Архитектура игрока

### Шаг 1. Базовый контракт игрока

Сейчас игрок уже реализован через `PlayerEntity`.

Текущие части, которые переиспользуем:

- `StatsEntityData`
  Источник stat values игрока.
- текущие movement actions
  Используются как база под управление от input data.

Под тестовое целевой состав данных игрока:

- `StatsEntityData`
  Источник `SPEED`, `HP`, `ATK`.
- `MoveInputData`
  Направление движения.
- `LookInputData`
  Направление взгляда или точка прицеливания.
- `FireInputData`
  Состояние выстрела.
- `PlayerHealthData`
  Текущее здоровье игрока, если решим хранить current/max отдельно.
- `PlayerCombatData`
  Cooldown, fire state, runtime данные стрельбы.

Минимально допустимый вариант:

- `Max HP` хранить в stat system;
- `Current HP` хранить в отдельном runtime data;
- урон и скорость брать из итоговых stat values.

### Шаг 2. Движение, поворот и стрельба

Что переиспользуем:

- существующий movement pipeline;
- текущий `Rigidbody`-based подход.

Что меняем:

- input должен писаться в `MoveInputData`, `LookInputData`, `FireInputData`;
- auto-attack flow не является целевым поведением тестового задания;
- нужен ручной `ShootAction`.

Алгоритм `ShootAction`:

1. Проверить `FireInputData`.
2. Проверить cooldown.
3. Взять aim direction или ray из камеры.
4. Выполнить `Physics.Raycast`.
5. Найти `EnemyEntity`.
6. Нанести урон на основе `ATK`.
7. Обновить cooldown.

### Шаг 3. HP игрока и применение статов

В тестовом игрок должен иметь:

- HP;
- отображение HP в UI;
- повышение HP через апгрейды.

Предлагаемый подход:

- stat `HP` отвечает за максимум;
- runtime health data отвечает за текущее значение;
- UI читает `Current HP` и `Max HP`;
- апгрейды меняют максимум через progression pipeline.

### Шаг 4. Проверки и критерии готовности игрока

Игрок готов, если:

- движение работает через WASD;
- стрельба работает по кнопке мыши;
- урон зависит от stat `ATK`;
- скорость зависит от stat `SPEED`;
- HP игрока отображается в UI;
- код игрока не завязан напрямую на desktop-only API.

## 3. Архитектура ввода игрока и подготовка под mobile

### Шаг 1. Разделение InputData на три независимых ComponentData

Главное изменение: общий input разделяется на отдельные данные:

- `MoveInputData`
- `LookInputData`
- `FireInputData`

Так gameplay будет читать только контракт данных, а не конкретную реализацию ввода.

### Шаг 2. Desktop input actions

Desktop input layer должен:

- читать WASD;
- читать mouse look или world aim;
- читать fire button;
- записывать результат в runtime input data.

Нужен отдельный источник:

- `DesktopPlayerInputSource`.

### Шаг 3. Mobile input actions

Для mobile прямо сейчас полноценный экранный input можно не реализовывать, но архитектура должна это позволять.

Нужна будущая точка расширения:

- `MobilePlayerInputSource`
  джойстик + экранная кнопка огня.

### Шаг 4. Контракт между input-слоем и gameplay

Контракт такой:

- input-слой пишет данные;
- gameplay только читает данные;
- `PlayerEntity` не знает, desktop это или mobile.

### Шаг 5. Проверки и критерии готовности input-архитектуры

Input-архитектура готова, если:

- gameplay не зависит от конкретного input backend;
- можно подменить источник ввода через DI;
- desktop input не требует переписывания movement и shooting logic при добавлении mobile.

## 4. Архитектура врагов

### Шаг 1. Что переиспользуем из текущего проекта

Переиспользуем:

- `EnemyEntity`;
- `EnemySpawnConfig`;
- текущий spawner;
- реестр врагов и сервисы респавна;
- `KillCounterService`.

### Шаг 2. Минимальный AI и границы сложности

Для тестового достаточно простого AI:

- ходить по waypoint path;
- не иметь сложной боевой логики;
- умирать при обнулении HP.

### Шаг 3. Контракт EnemyEntity: ComponentData и Actions

У врага должны быть:

- stat или runtime HP;
- action получения урона;
- action смерти;
- регистрация в реестре;
- визуальная деактивация или уничтожение.

### Шаг 4. Случайное стартовое HP врагов

Стартовое HP задается через конфиг.

Допустимый вариант:

- диапазон стартового HP в `EnemySpawnConfig`;
- при спавне каждому врагу назначается случайное значение.

### Шаг 5. Спавн через WaypointPathEntitySpawner

Переиспользуем существующий pipeline спавна врагов и не строим новый spawner с нуля.

### Шаг 6. Связь смерти врага и награды игроку

После смерти врага:

1. Увеличивается `KillCounterService`.
2. Progression layer начисляет 1 `upgrade point`.
3. UI автоматически получает обновление.

### Шаг 7. Проверки и критерии готовности врагов

Враги готовы, если:

- спавнятся через текущий сервисный слой;
- получают урон от выстрела;
- умирают при окончании HP;
- начисляют награду игроку;
- не требуют сложной ручной настройки в сцене для каждого экземпляра.

## 5. Архитектура улучшений статистик и конфиги

Этот раздел нужно реализовать как расширение `Stats` asmdef и связанного runtime-пайплайна игрока. По мотивам `GameCreator Tables` таблица должна быть универсальным конфигом: сама таблица ничего не знает про игрока, опыт или конкретную характеристику. Она умеет работать с прогрессией как с функцией `уровень <-> накопленное значение`, а `StatsEntityData` и gameplay уже решают, как именно использовать эту функцию.

### Шаг 1. Подготовить stat assets, универсальные таблицы и runtime-компоненты stat

Нужно подготовить stat assets игрока:

- `SPEED`
- `HP`
- `ATK`
- `EXP`

Также нужно добавить универсальный конфиг таблицы. Идея такая же, как в `GameCreator`: таблица хранит progression-данные и дает единое API для вычисления уровня, значения уровня и прогресса между уровнями.

Минимальные требования к универсальному `Table`:

- это `ScriptableObject`-конфиг без привязки к игроку;
- может хранить progression либо списком значений, либо через формульный режим;
- работает с накопленным значением, а не только с прямым индексом;
- имеет единый API для всех сценариев: level-up по `EXP`, рост `HP`, рост `SPEED`, рост `ATK`.

Минимальный API таблицы:

- `float GetValue(int level)`
- `float GetDelta(int level)`
- `int GetLevel(float value)`
- `float GetPreviousValue(int level)`
- `float GetNextValue(int level)`
- `float GetProgress01(float value)`
- `int ClampLevel(int level)`
- `int MaxLevel`

Смысл API:

- `GetValue(level)` возвращает накопленное значение на указанном уровне;
- `GetDelta(level)` возвращает прирост относительно предыдущего уровня;
- `GetLevel(value)` возвращает уровень по накопленному значению;
- `GetProgress01(value)` нужен для UI и future use, если позже захотим показывать прогресс до следующего уровня.

Для `SPEED`, `HP`, `ATK` в `Stat.ComponentStack` добавляется authoring-компонент, который ссылается на таблицу и умеет создать runtime-компонент для `RuntimeStat`. Он вешается именно на `Stat`, а не на `Entity`.

```csharp
public interface IRuntimeStatComponent
{
}

public interface IRuntimeStatComponentFactory
{
    IRuntimeStatComponent CreateRuntimeComponent(RuntimeStatBuildContext context);
}

[Name("Stats/Table")]
[Group("Stats")]
public sealed class StatTableComponent : ComponentData, IRuntimeStatComponentFactory
{
    [OdinSerialize] private Table _table;
    [OdinSerialize] private int _initialLevel;

    public Table Table => _table;

    public IRuntimeStatComponent CreateRuntimeComponent(RuntimeStatBuildContext context)
    {
        int level = PlayerPrefs.GetInt($"{context.StatId}Level", _initialLevel);
        return new RuntimeStatLevelComponent(_table, _table.ClampLevel(level));
    }
}
```

`RuntimeStatBuildContext` должен содержать минимум:

- `string StatId`
- при необходимости ссылку на `StatsEntityData` или `RuntimeStat`, если runtime-компоненту нужен доступ к соседнему state

`RuntimeStat` нужно расширить runtime-компонентами:

- добавить контейнер runtime-компонентов;
- добавить `GetRuntimeComponent<T>()`;
- добавить `TryGetRuntimeComponent<T>(out T component)`;
- дать `Stat.ComponentStack` возможность создавать runtime-компоненты при сборке stat.

Минимальный набор новых runtime-типов:

```csharp
[Serializable]
public sealed class RuntimeStatLevelComponent : IRuntimeStatComponent
{
    [OdinSerialize] private Table _table;
    [OdinSerialize] private ReactiveProperty<int> _appliedLevel = new(0);

    public Table Table => _table;
    public ReactiveProperty<int> AppliedLevel => _appliedLevel;

    public RuntimeStatLevelComponent(Table table, int appliedLevel)
    {
        _table = table;
        _appliedLevel.Value = appliedLevel;
    }
}
```

Цель этого шага:

- не создавать отдельный `StatLevelsEntityData`;
- хранить applied level внутри `RuntimeStat`;
- позволить `Stat.ComponentStack` не только описывать stat, но и участвовать в создании runtime-модели;
- сделать `Table` универсальным механизмом, который переиспользуется и для прогрессии игрока, и для роста боевых статов.

На этом шаге также нужно обновить `PlayerEntityCollection`:

- добавить `EXP`;
- добавить `HP`, если его еще нет в коллекции игрока;
- убедиться, что у боевых stat есть `StatValueComponent` с base value;
- повесить `StatTableComponent` на `SPEED`, `HP`, `ATK`;
- не вешать `StatTableComponent` на `EXP`, потому что `EXP` не является улучшаемой боевой характеристикой.

### Шаг 2. Расширить StatsEntityData под уровни stat и сохранение runtime-state

Текущий [StatsEntityData.cs](/C:/Users/Vlad/ShooterUpgradePrototype/Packages/com.vladislavtsurikov.frameworks/Stats/EntityDataActionIntegration/ComponentData/StatsEntityData.cs) на `RebuildFromCollection()` полностью пересоздает словарь `_stats`. В этом виде он теряет runtime state и легко ломает реактивные подписки на `RuntimeStat.Value`.

Нужно расширить `StatsEntityData` в четырех местах:

1. На этапе `RebuildFromCollection()` создавать runtime-компоненты из `Stat.ComponentStack`.
2. До пересборки сохранять runtime value для stat с `PreserveRuntimeValueComponent`.
3. До пересборки сохранять applied level для stat с `RuntimeStatLevelComponent`.
4. По возможности переиспользовать существующие `RuntimeStat` по `statId`, чтобы не рвать UniRx-подписки.

Минимальный API, который нужно добавить:

- `T GetRuntimeComponent<T>(string statId) where T : class, IRuntimeStatComponent`
- `bool TryGetRuntimeComponent<T>(string statId, out T component) where T : class, IRuntimeStatComponent`
- `int GetStatLevelById(string statId)`
- `bool SetStatLevelById(string statId, int level)`
- `bool AddStatLevelById(string statId, int delta)`

Логика `RebuildFromCollection()` должна быть такой:

```csharp
public void RebuildFromCollection()
{
    Dictionary<string, float> preservedValues = CapturePreservedValues();
    Dictionary<string, int> preservedLevels = CaptureStatLevels();

    foreach (Stat stat in _collection.Stats)
    {
        RuntimeStat runtimeStat = GetOrCreateRuntimeStat(stat);
        runtimeStat.SetBaseValue(GetDefaultValue(stat));
        runtimeStat.ClearRuntimeComponents();

        BuildRuntimeComponents(stat, runtimeStat, preservedLevels);

        if (preservedValues.TryGetValue(stat.Id, out float preservedValue))
        {
            runtimeStat.CurrentValue = ApplyClamp(stat, preservedValue);
        }
    }

    RemoveMissingStats();
}
```

Для `EXP` нужен отдельный authoring-компонент `PreserveRuntimeValueComponent`. Его задача одна: отметить stat, чье текущее runtime-значение должно переживать `RebuildFromCollection()`. Этот компонент вешается на `EXP`.

### Шаг 3. Реализовать EXP внутри Stats и расчет уровня игрока

Никакого отдельного `ExperienceData` не нужно. `EXP` остается обычным stat в `StatsEntityData`, чтобы progression жил в том же stat-domain, что и остальная система.

Технически нужно сделать следующее:

- добавить `EXP` в `PlayerEntityCollection`;
- повесить на `EXP` `StatValueComponent` и `PreserveRuntimeValueComponent`;
- добавить конфиг прогрессии игрока, который ссылается на универсальный `Table` опыта;
- добавить `PlayerProgressionService`, который работает через `StatsEntityData`;
- хранить текущий доступный пул очков в `UpgradePointsData`;
- сохранять `EXP` и `UpgradePointsData` через `PlayerPrefs`.

Pipeline начисления опыта:

1. Смерть врага публикует награду в progression-слой.
2. `PlayerProgressionService` вызывает `StatsEntityData.AddStatValueById("EXP", amount)`.
3. Сервис читает новое значение `EXP`.
4. Через универсальный `Table.GetLevel(exp)` вычисляет новый уровень игрока.
5. Если уровень вырос, начисляет разницу уровней в `UpgradePointsData`.
6. После изменения `EXP` и `UpgradePointsData` новые значения пишутся в `PlayerPrefs`.

Уровень игрока не нужно хранить как отдельный source of truth. Его можно кэшировать в `PlayerProgressionService`, но после загрузки он всегда восстанавливается из `EXP` через таблицу.

Правила ключей:

- `EXP` сохраняется как `EXPValue`
- доступные очки сохраняются как `UpgradePoints`

Опорная логика сохранения прогрессии должна быть такой:

```csharp
public void AddExperience(float amount)
{
    _statsEntityData.AddStatValueById("EXP", amount);

    float exp = _statsEntityData.GetStatValueById("EXP");
    int newLevel = _playerLevelTable.GetLevel(exp);
    int levelDelta = newLevel - _cachedLevel;

    if (levelDelta > 0)
    {
        _upgradePoints.Value += levelDelta;
        _cachedLevel = newLevel;
        PlayerPrefs.SetInt("UpgradePoints", _upgradePoints.Value);
    }

    PlayerPrefs.SetFloat("EXPValue", exp);
    PlayerPrefs.Save();
}
```

Восстановление прогрессии должно быть таким:

1. `StatsEntityData` строит runtime stat из `StatCollection`.
2. Для `EXP` после `RebuildFromCollection()` читается `PlayerPrefs.GetFloat("EXPValue", defaultExp)`.
3. Значение `EXP` записывается в `StatsEntityData.SetStatValueById("EXP", savedExp)`.
4. `UpgradePointsData` получает значение из `PlayerPrefs.GetInt("UpgradePoints", 0)`.
5. `PlayerProgressionService` пересчитывает текущий уровень через `Table.GetLevel(savedExp)`.

В результате applied levels stat, `EXP` и `UpgradePoints` все сохраняются в одном простом стиле через ключи `PlayerPrefs`, без отдельного save DTO и без отдельного storage-сервиса.

### Шаг 4. Реализовать применение уровней stat через таблицы

В окне улучшений нужен отдельный draft-state:

- `PendingUpgradeState`

Он хранит только временные изменения по `StatId`, пока окно открыто. После нажатия `Apply` черновик переносится в applied levels внутри `StatsEntityData`.

Для применения уровней нужен отдельный action:

- `ApplyStatLevelsByTableAction`

Его задача:

1. Пройти по всем runtime stat игрока.
2. Найти stat с `RuntimeStatLevelComponent`.
3. Прочитать `AppliedLevel`.
4. Взять из `Table` итоговое значение `GetValue(level)`.
5. Сравнить его с `StatValueComponent.BaseValue`.
6. Построить runtime upgrade-эффект с нужной дельтой.
7. Полностью пересобрать `ModifiersData` игрока.
8. Передать управление в `ApplyModifierStatEffectAction`.

Опорный код действия должен выглядеть так:

```csharp
foreach (RuntimeStat runtimeStat in statsEntityData.Stats.Values)
{
    if (!runtimeStat.TryGetRuntimeComponent(out RuntimeStatLevelComponent levelComponent))
    {
        continue;
    }

    float baseValue = runtimeStat.Stat.ComponentStack.GetElement<StatValueComponent>().BaseValue;
    float targetValue = levelComponent.Table.GetValue(levelComponent.AppliedLevel.Value);
    float delta = targetValue - baseValue;

    effects.Add(BuildModifier(runtimeStat.Stat.Id, delta));
}
```

Что нужно реализовать рядом с этим action:

- `PlayerUpgradeService`, который переносит значения из `PendingUpgradeState` в `StatsEntityData.SetStatLevelById(...)`;
- единый метод `ModifiersData.ReplaceAll(...)`, чтобы не накапливать старые upgrade-эффекты;
- runtime-представление upgrade-эффекта, если окажется, что `ModifierStatEffect` слишком сильно завязан на asset workflow;
- подписку на изменения applied level или явный вызов `ApplyStatLevelsByTableAction` после `Apply`.

UI не должен напрямую писать итоговые значения `SPEED`, `HP`, `ATK`. UI меняет только pending levels, а итоговые stat values всегда пересчитываются через таблицу и modifier pipeline.

### Шаг 5. Реализовать сохранение и восстановление прогресса

В этом проекте сохранение прогресса можно сделать максимально простым и прямым через `PlayerPrefs`, без отдельного DTO и без общего save snapshot. Нам нужно сохранять только applied level прокачиваемых stat, поэтому достаточно сохранять каждый уровень по отдельному ключу, построенному из `Stat.Id`.

Правило ключа:

- ключ строится как `"{StatId}Level"`
- примеры:
  - `SPEEDLevel`
  - `HPLevel`
  - `ATKLevel`

Что сохраняем:

- applied level для каждого stat с `RuntimeStatLevelComponent`

Что не сохраняем:

- `PendingUpgradeState`
- runtime modifiers
- итоговые пересчитанные значения `SPEED`, `HP`, `ATK`
- временные UI-состояния

Идея сохранения:

1. Игрок подтверждает изменения через `Apply`.
2. `PlayerUpgradeService` или `StatsEntityData.SetStatLevelById(...)` обновляет applied level.
3. В этот же момент вызывается `PlayerPrefs.SetInt($"{statId}Level", appliedLevel)`.
4. После записи вызывается `PlayerPrefs.Save()`.

Сохранять нужно именно в точке изменения applied level, а не отдельным проходом по всем stat. Тогда save-flow получается автоматическим: любой stat, который поменял уровень, сразу записывает свой persisted state.

Идея восстановления:

1. `StatsEntityData` начинает строить runtime stat из `StatCollection`.
2. `Stat.ComponentStack` создает runtime-компоненты через `BuildRuntimeData`.
3. Если у stat есть `StatTableComponent`, он пытается прочитать сохраненный уровень из `PlayerPrefs`.
4. Если ключ найден, используется сохраненный уровень.
5. Если ключа нет, используется стартовый уровень из authoring-конфига, например `_initialLevel`.
6. После сборки всех runtime stat запускается `ApplyStatLevelsByTableAction`.
7. `ApplyModifierStatEffectAction` пересчитывает итоговые значения stat.

Опорная логика внутри `StatTableComponent` должна быть такой:

```csharp
public IRuntimeStatComponent CreateRuntimeComponent(RuntimeStatBuildContext context)
{
    string key = $"{context.StatId}Level";
    int savedLevel = PlayerPrefs.GetInt(key, _initialLevel);
    int level = _table.ClampLevel(savedLevel);

    return new RuntimeStatLevelComponent(_table, level);
}
```

Опорная логика сохранения уровня должна быть такой:

```csharp
public bool SetStatLevelById(string statId, int level)
{
    RuntimeStatLevelComponent component = GetRuntimeComponent<RuntimeStatLevelComponent>(statId);
    int clampedLevel = component.Table.ClampLevel(level);

    if (component.AppliedLevel.Value == clampedLevel)
    {
        return false;
    }

    component.AppliedLevel.Value = clampedLevel;
    PlayerPrefs.SetInt($"{statId}Level", clampedLevel);
    PlayerPrefs.Save();
    MarkDirty();
    return true;
}
```

Преимущества такого подхода:

- не нужны `PlayerProgressSaveData` и `StatLevelSaveData`;
- не нужен отдельный storage abstraction для этого тестового;
- добавление нового прокачиваемого stat не требует менять save schema;
- `BuildRuntimeData` сам подхватывает сохраненный уровень;
- логика остается data-driven: ключ строится из `Stat.Id`, а не хардкодится под `Speed/HP/ATK`.

Ограничения подхода:

- он сохраняет только applied levels stat;
- если позже захотим хранить больше прогресса, возможно придется вынести save-слой в отдельный сервис;
- `PlayerPrefs.Save()` лучше вызывать не слишком часто, но для редких нажатий `Apply` в тестовом это допустимо.

### Шаг 6. Draft и Apply flow

Это главный функциональный сценарий UI.

Открытие окна:

1. Игрок нажимает кнопку на HUD.
2. Открывается `UpgradeWindowHandler`.
3. Создается draft-копия applied state.
4. UI заполняется значениями из конфига и applied state.
5. Для каждой характеристики собираются сегменты: `filled = applied level + pending`, `empty = max level - filled`.

Нажатие `+`:

1. Проверяется, есть ли свободные draft points.
2. Проверяется, не достигнут ли `MaxLevel`.
3. Увеличивается только draft level.
4. Уменьшаются только draft points.
5. В строке обновляются сегменты и `pendingDeltaLabel`.
6. Реальные характеристики игрока не меняются.

Нажатие `Apply`:

1. Draft state передается в `PlayerUpgradeService`.
2. Сервис обновляет applied state.
3. `PlayerUpgradeEffectResolver` пересобирает modifiers.
4. `ModifiersData` обновляется.
5. Игрок получает новые итоговые stat values.
6. HUD автоматически отражает новый state.
7. Окно закрывается.

Дополнительные UI-правила:

- `Apply` может оставаться активной всегда, но безопаснее выключать ее при отсутствии pending-изменений;
- кнопка `+` блокируется, если `draft points == 0` или достигнут `MaxLevel`;
- закрытие по `X` и клик по backdrop должны работать одинаково: отмена draft без применения.

Нажатие `X`:

1. Draft state уничтожается.
2. Applied state остается прежним.
3. Окно закрывается.
